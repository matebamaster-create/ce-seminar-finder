# 7. 実装フェーズ分解

## 進め方

各Phaseは、管理者が原則「OK」と返せば次へ進められる単位にしています。APIキー、Googleアカウント共有、独自ドメイン等、外部設定が必要になる時点だけ、まとめて確認事項を提示します。

- 要件要約と設計論点
- 対象9団体のサイト調査
- 技術構成6案の比較
- スプレッドシート・データ設計
- スクレイピング・AI・重複・ReviewQueue設計

完了条件:

- 推奨構成と主要データ境界を管理者が確認
- 次Phaseで作るものが明確

3. 13シートの作成スクリプト

5. 共通データ型、ID、日時、状態遷移

- 秘密をリポジトリへ置かない

Phase 1開始時に必要なもの:

状態: **完了（沖縄は設計どおり技術的保留）**

各サイトについて保存する検証サンプル:

- 一覧HTMLまたはフィード
- 詳細HTML
- PDFリンク例
- イベント候補と対象外候補

完了条件:

- 9団体すべてが成功、部分成功、技術的保留のいずれかで明示される

沖縄で公開APIが確認できない場合:

- それも不安定なら`SOURCE_GAP`として、日臨工等の補助ソースから候補を保存する

1. PDF差分取得、文字抽出、品質評価、チャンク保存

3. AIProviderインターフェース

5. 構造化出力スキーマと検証
6. イベント性、形式、種別、ジャンル、主催形態の判定

8. 入力ハッシュによるAIキャッシュ
9. プロンプトインジェクション耐性テスト

完了条件:

- サンプルHTML/PDFから共通スキーマを生成

Phase 3開始時に必要なもの:

- 利用するAI APIの選択とキー
- 月次予算の警告額

キーが未準備でも、NoopProviderと保存済みサンプルで先に実装・テストできます。

## 全国展開前の移行判定

次のいずれかに達したらSheetsからDBへの移行を検討します。

- 1,000万セルの30%を超える見込み
- 日次処理がActionsの許容時間・無料枠を恒常的に圧迫
- 管理者が複数人になり同時編集・承認権限が必要
- ユーザー登録、通知、お気に入りが必要
- PDF検索インデックスがブラウザで重くなる

## 今回採用した前提

- 公開コードでも問題ないものとして、無料のGitHub Pagesを優先
- Sheetsは非公開
- 限定共有は認証ではなく`noindex`＋URL共有
- 最初のAIプロバイダーはPhase 3開始時にAPIキーの用意があるものを採用
- ムームーサーバーはMVP公開先に使わず、将来の動的機能候補として保留

## 主なリスク

| リスク | 対応 |
|---|---|
| サイト構造変更 | アダプター分離、失敗ログ、保存サンプルテスト |
| 不確かな料金・単位の誤公開 | 厳格なReviewQueue、公式リンク優先 |
| PDF解析精度 | 品質スコア、根拠、OCR限定、要確認 |
| 同一イベントの誤統合 | 決定的アンカーなしの自動統合禁止 |
| APIコスト増 | ハッシュキャッシュ、上限、曖昧候補だけAI |
| Sheets容量 | チャンク化、監視、全国展開前のDB判定 |
| Actions/GASクォータ | 差分処理、分割、実行時間監視 |
| 取得拒否・削除依頼 | Sources即時停止、最小転載、連絡窓口 |
| 旧ドメインの第三者利用 | 許可ホスト、公式性の定期再確認 |
| 公開リポジトリへの漏えい | 公開列allowlist、秘密スキャン、ビルド検査 |

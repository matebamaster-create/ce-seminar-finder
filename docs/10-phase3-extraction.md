# 10. Phase 3 PDF・AI抽出基盤

実装・確認日: 2026年7月4日

## 状態

APIキーを必要としないPhase 3の基盤を実装しました。PDF処理、ルール抽出、AIプロバイダー境界、構造化結果の検証、入力ハッシュキャッシュ、プロンプトインジェクション対策を利用できます。

実AI APIは呼び出していません。APIの選択、キー、月次予算が未設定の間はNoopProviderが候補を`AI_PENDING`へ送り、収集データを失わず課金もしません。

## PDF処理

- PDF署名、MIME、20MB上限を検査
- 暗号化PDFを拒否
- 文字層をページ単位で抽出
- 文字量、置換文字、制御文字から品質を0〜1で評価
- 低品質・空ページをOCR候補として記録
- 「重要候補」かつOCR実装が設定された場合だけ対象ページをOCR
- ページ境界を付けて最大40,000文字に分割
- 文書とチャンクにSHA-256を付与
- PDF全文・生チャンクは公開成果物へ含めない

実PDFを2ページ生成して確認し、文字層抽出は2ページ、状態`success`、品質0.875でした。1ページ目を画像化して、文字の欠落、重なり、クリップがないことも確認しました。検証用PDFと画像は完了後に削除しています。

## 決定的ルール

AIより先に、原文根拠付きで次を抽出します。

- タイムゾーン付き開催日時
- 参加費の原文と無料／有料
- 単位・ポイントの原文
- 入力に実在し、許可リストと一致する申込URL

ルール値もAI値と同じ`ExtractedField`形式で、値、信頼度、300文字以下の根拠を持ちます。

## AI境界

`AIProvider`はプロバイダー名、モデル名、`extract_event`だけを要求します。アプリ本体はベンダー固有レスポンスを扱いません。

現在の実装:

- `NoopProvider`: 課金なし。`AI_PENDING`へ送る
- `ExtractionService`: 呼び出し、検証、キャッシュを統括
- `JsonExtractionCache`: 入力、モデル、プロンプト版のSHA-256で再利用

プロンプト版、候補本文、PDFチャンク、ルール値のいずれかが変わると別キャッシュになります。

## 検証ルール

AIの自己申告をそのまま保存せず、次を機械検査します。

- 信頼度を0〜1へ制限
- 未知のフィールドを破棄
- 開催形式、種別、主催形態、料金区分をマスタ照合
- 日時はISO 8601かつタイムゾーン必須
- 日付、料金、単位は根拠がなければ`null`
- 公式・申込URLは入力中の許可URLとの完全一致が必要
- AIが作ったURLは`URL_NOT_IN_SOURCE`として破棄
- 不正値は理由コードを付け、候補全体は失わない

## プロンプトインジェクション対策

候補HTMLやPDFの文章は`SOURCE_DATA`内の信頼できない引用として渡します。システム指示では、引用内の命令、依頼、役割変更に従わないこと、URLを作らないこと、根拠のない値を`null`にすることを固定します。

これはモデルへの注意書きだけに依存せず、出力後のURL許可リストと高リスク項目の根拠検査でも強制します。

## ローカル実行

PDF依存関係を含めてインストール:

```bash
python3 -m pip install -e '.[pdf]'
```

非公開の解析結果を出力:

```bash
python3 -m ce_seminar_finder.cli extract-pdf \
  --input /path/to/event.pdf \
  --output var/extraction/event.json
```

出力にはPDF本文チャンクが含まれるため、`var/`等の非公開・Git対象外領域だけを指定します。

## 検証結果

全48件のテストが通過しています。Phase 3では次を追加確認しました。

- 不正署名、MIME、サイズ超過の拒否
- ページ品質、チャンク、ハッシュ
- 重要候補だけの限定OCR
- URL捏造の破棄
- 根拠なしの日付・料金・単位の破棄
- 未知の分類とタイムゾーンなし日時の破棄
- ルール抽出
- ページ内の命令文を引用へ閉じ込めるプロンプト
- プロンプト版・入力変更時のキャッシュ無効化
- 同一入力のAI呼び出し再利用
- NoopProviderの`AI_PENDING`

## 次の外部設定

実AI接続時に必要なのは、利用API、APIキー、月次警告予算です。設定されるまで実呼び出しを行わず、Phase 4以降の管理機能を先行できます。
